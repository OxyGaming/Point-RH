Présentation de Point RH

Système de simulation de remplacement pour les journées de service SNCF

Point RH est un outil d'aide à la décision permettant aux encadrants de trouver rapidement quel agent peut remplacer un collègue absent sur une Journée de Service (JS) imprévisible, en vérifiant automatiquement toutes les règles réglementaires (repos, amplitude, GPT, TE…).

Simulation Multi-JS

Une ou plusieurs JS à couvrir. Le système propose des répartitions globales optimisées, avec résolution en cascade si nécessaire, sur 4 scénarios parallèles.

Mode principal

Fiches agents

Profil complet de chaque agent : habilitations, LPA, régime, règles de déplacement personnalisées.

Import planning

Chargement des plannings depuis un fichier Excel/TXT exporté du système RH source.

Glossaire — Définitions des termes

Tous les termes utilisés dans Point RH, avec leur définition précise

Termes fondamentaux

JS
Journée de Service
Un poste de travail planifié, défini par une date, une heure de début, une heure de fin, un code (ex. PEY006R) et un type. C'est l'unité de base du planning.
Ex : JS du 17 mars, 21h00–05h00, code GIC004R
NPO
Non Présence Opérationnelle
Toute absence planifiée : congés, repos hebdomadaire (RU), repos périodique (RP), maladie, inaptitude, formation. Ces événements figurent dans le planning mais ne sont pas des journées de service.
Ex : Congé du 20 au 22 mars, repos périodique le 18 mars
GPT
Grande Période de Travail
Séquence de journées de service consécutives entre deux Repos Périodiques (RP). Maximum 6 JS par GPT. Détermine le cumul TE, la nature nuit/jour, et déclenche les vérifications de repos périodiques.
Ex : Lundi JS, Mardi JS, Mercredi JS, Jeudi RP → GPT de 3 jours
TE
Travail Effectif
Durée réelle de travail d'une journée de service, hors temps de trajet. Limite par JS : 10h standard, 8h30 de nuit, 12h en remplacement. Cumul maximum sur une GPT : 48h.
JS de 21h00 à 05h00 = amplitude 8h, TE = 7h30 (si 30min de trajet)
RP
Repos Périodique
Période de repos réglementaire qui clôture une GPT. Trois niveaux : simple (36h), double (60h), triple (84h). Une GPT > 5 JS doit être suivie d'un RP double minimum.
RP simple : partir jeudi soir, reprendre samedi matin (36h gap)
LPA
Lieu Principal d'Affectation
La base géographique de référence d'un agent. Si une JS se déroule hors LPA, l'agent est en déplacement et des temps de trajet s'ajoutent à son amplitude effective.
LPA = Givors. JS à Lyon Part-Dieu → hors LPA → déplacement
Amplitude
Amplitude de service
Durée totale d'une journée de service, du début effectif à la fin effective. Inclut les temps de trajet si l'agent est en déplacement. Limite : 11h standard, jusqu'à 13h en déplacement + remplacement.
JS 20h00–06h10 = amplitude 10h10. Avec trajet +1h = amplitude effective 11h10
Repos journalier
Repos entre deux postes
Durée minimale obligatoire entre la fin d'un poste et le début du suivant. Minimum 12h standard, 14h après un poste de nuit, 10h pour un agent de réserve (1 fois/GPT).
Fin JS le 17 mars à 06h10. Prochain poste le 17 mars à 20h00 → repos = 13h50 ✓

Termes de la simulation

Conforme
Toutes les règles réglementaires sont respectées. L'agent est mobilisable sans réserve. Badge vert dans l'interface.
Vigilance
Une règle non-bloquante est proche de son seuil ou légèrement dépassée. L'agent peut être mobilisé avec réserve. L'encadrant doit valider manuellement. Badge jaune.
Non-conforme
Une ou plusieurs violations bloquantes existent. L'agent ne peut pas prendre ce poste. Badge rouge.
Cascade
Mécanisme de résolution en chaîne. Si l'agent A est bloqué par un conflit repos (sa JS B l'en empêche), on cherche un agent C pour reprendre la JS B, libérant ainsi A. Jusqu'à 10 niveaux.
A doit faire JS1 mais bloqué par JS2 → C reprend JS2 → A libre pour JS1
Figeage
Action de "geler" une JS de type DERNIER_RECOURS pour libérer l'agent planifié dessus, afin qu'il puisse prendre un imprévu plus urgent. La JS figée reste non-couverte.
JS Z / JS de type Z
Journée de service sans charge réelle (placeholder). Un agent prévu sur une JS Z peut être réaffecté sans avoir besoin de trouver un remplaçant pour cette JS d'origine. Bonus +15 pts au score.

Quatre familles sont assimilées JS Z :
  • Suffixe « Z » sur le code JS — ex. GIV Z, GIC Z, PEY Z
  • Préfixe « FO » sur le code JS (formation) — ex. FO123, FOX
  • typeJs = « DIS » (famille Disponibilité, sans charge) — quel que soit le code JS
  • Préfixe additionnel configuré par l'admin dans la page Préfixes JS Z — ex. DISP, PSE, FORM (au cas par cas, pour couvrir des codes locaux)
JS FO
Journée de service de type Formation/Organisation. Identifiée par le préfixe "FO". Peut être réaffectée comme une JS Z si l'agent est mobilisé sur un imprévu.
DIS (typeJs)
Famille de journée de service marquée comme sans charge réelle dans le SI source (colonne TYPE JS / FAM. NPO). Toute ligne dont le typeJs vaut « DIS » est traitée par le moteur exactement comme une JS Z : pas de cascade, +15 pts au score, badge violet « JS Z » à l'affichage.
Préfixes JS Z additionnels
Liste configurable par l'admin (page Admin → Préfixes JS Z) pour assimiler à JS Z des codes JS locaux ou exotiques non couverts par les règles built-in. Chaque entrée est un préfixe : DISP couvre DISP, DISP01, DISPO_FORMATION… Effet identique aux autres familles JS Z. Idéal pour ne pas hardcoder la liste et l'ajuster sans déploiement.
JsType
Référentiel central d'un type de journée de service. Définit les horaires standards (indépendants des trajets), la durée, si c'est une JS de nuit, le régime, et la flexibilité (OBLIGATOIRE ou DERNIER_RECOURS).
JsType "PEY006R" : 21h00–05h00, 8h, estNuit=true, OBLIGATOIRE
OBLIGATOIRE
Flexibilité d'un JsType. La JS doit être couverte. Pénalité maximale si non-couverte en simulation multi-JS.
DERNIER_RECOURS
Flexibilité d'un JsType. La JS peut rester non-couverte avec une pénalité réduite. L'agent planifié peut être libéré par figeage pour couvrir une JS OBLIGATOIRE plus urgente.
Déplacement
Situation où une JS se déroule hors du LPA de l'agent. Des temps de trajet (aller/retour) s'ajoutent automatiquement à l'amplitude. Nécessite l'habilitation peutEtreDeplace.
Régime B / Régime C
Régimes réglementaires spéciaux imposant un Travail Effectif minimum de 5h30 par journée de service. Les agents en Régime B ou C ne peuvent pas prendre de postes trop courts.
Agent de réserve
Agent dédié aux remplacements. Bénéficie de règles légèrement assouplies : repos minimum 10h (au lieu de 12h) une fois par GPT, amplitude nuit réduite à 10h30. Bonus +10 pts au score.

Termes métier complémentaires

APES
Affectation Principale d'Emploi Spécifique
Code identifiant le poste principal d'un agent dans la nomenclature SNCF.
CTF
Code du Travail Ferroviaire
Cadre réglementaire français qui régit les temps de travail, repos, et conditions d'emploi dans le ferroviaire. Source de tous les seuils par défaut de Point RH.
DJS
Détenteur de Journée de Service
Agent normalement titulaire du poste. Si un remplaçant prend sa place (case « Remplacement » cochée), des plafonds spéciaux s'appliquent (TE +2h, amplitude jusqu'à 13h avec déplacement).
UCH
Unité de Conduite Homologuée
Unité organisationnelle de rattachement d'un agent. Utilisée pour filtrer la liste des agents et identifier le périmètre d'intervention.
RU
Repos Unique (ou universel)
Type de repos réglementaire interne à une GPT. Ne constitue pas un RP — n'interrompt pas la GPT.
Motif principal
Raison principale affichée pour expliquer le statut d'un agent (ex. « Repos journalier insuffisant : 11h30 < 12h »). Concatène la violation la plus critique en cas de multiples violations.
Indéterminable
Calcul qu'on ne peut pas effectuer automatiquement faute de configuration (LPA manquant, type de JS non référencé). Nécessite une saisie manuelle côté simulation ou une correction admin.
Préfixe JS
Partie du code JS identifiant le type de poste (ex. « GIV » dans « GIV023 »). Les habilitations d'un agent sont stockées sous forme de liste de préfixes.
Score de pertinence
Note 0–100 indiquant la qualité d'une affectation. 100 = optimal. Basé sur conformité + marges + bonus réserve + bonus GPT 0 JS.
Portée
Périmètre des agents analysés dans une simulation multi-JS : « Tous » ou « Réserve uniquement ».
Robustesse
Niveau de sécurité réglementaire d'un scénario multi-JS : HAUTE (toutes affectations CONFORMES), MOYENNE (au moins une VIGILANCE), FAIBLE (plusieurs dérogations).
Soft delete
Suppression logique — l'entité n'est plus visible dans les listes mais reste en base pour assurer la traçabilité historique.

Termes techniques

JWT
JSON Web Token
Jeton sécurisé signé avec une clé secrète, porté par un cookie HTTP-only. Durée de validité 8h. Contient l'id utilisateur et le rôle.
HTTP-only
Cookie sécurisé
Cookie inaccessible au JavaScript du navigateur — protège contre le vol de session par XSS.
HTTPS
HTTP Sécurisé
Version chiffrée du protocole HTTP. Les échanges navigateur/serveur sont illisibles en transit. Obligatoire en production.
ORM
Object-Relational Mapping
Couche logicielle entre le code applicatif et la base de données. Point RH utilise Prisma qui génère les requêtes SQL à partir du schéma TypeScript.
RGPD
Règlement Général sur la Protection des Données
Cadre européen (2018) régissant le traitement des données personnelles. Point RH s'y conforme via minimisation, accès restreint, traçabilité et chiffrement.
CSP
Content Security Policy
En-tête HTTP qui restreint les sources de contenu autorisées (scripts, styles). Protection contre l'injection XSS.

Règles réglementaires

Les 10 règles vérifiées automatiquement par le moteur de simulation (8 bloquantes + 2 vigilance)

⚠ Règles bloquantes vs. vigilance
Une règle BLOQUANTE empêche la mobilisation. Une règle VIGILANCE avertit mais permet la mobilisation avec validation encadrant.
Bloquant
1. AMPLITUDE — Durée totale de la journée
L'amplitude (durée totale du début à la fin, trajet inclus) ne doit pas dépasser le seuil autorisé. La limite varie selon le contexte :
Standard : 11h (660 min) Nuit : 11h (660 min) Réserve + nuit : 10h30 (630 min) Déplacement : 12h (720 min) Déplacement + remplacement : 13h (780 min)
Bloquant
2. REPOS JOURNALIER — Repos entre deux postes
La durée entre la fin du dernier poste et le début de l'imprévu doit respecter un minimum. Varie selon le contexte du dernier poste :
Standard : 12h (720 min) Après nuit : 14h (840 min) Réserve + remplacement (1×/GPT) : 10h (600 min)
⚠ Si TE > 6h sans coupure interne, ajouter +20 min au repos minimum requis.
Bloquant
3. TRAVAIL EFFECTIF — Durée de travail par JS
Le travail effectif (hors trajet) par journée ne peut pas dépasser :
Standard : 10h (600 min) Nuit : 8h30 (510 min) Remplacement : 12h (720 min)
Bloquant
4. TE GPT 48H — Cumul travail effectif sur la GPT
Le cumul des TE de toutes les JS dans la GPT en cours plus la JS imprévisible ne doit pas dépasser 48 heures (2880 min).
💡 Exemple
GPT en cours : JS1=8h + JS2=8h30 + JS3=9h = 25h30 cumulés.
Imprévu = 8h. Total = 33h30 ✓
Si GPT avait déjà 45h cumulés + imprévu 5h = 50h → VIOLATION
Bloquant
5. GPT_MAX — Longueur maximale de la GPT
Une GPT ne peut pas dépasser 6 journées de service consécutives. Le système injecte la JS simulée dans le planning pour calculer la GPT résultante.
💡 Exemple
Agent en GPT de 5 JS. Ajouter 1 JS → GPT = 6 ✓
Agent en GPT de 6 JS. Ajouter 1 JS → GPT = 7 → VIOLATION
Bloquant
6. HABILITATION NUIT — Autorisation postes de nuit
Si la JS est une JS de nuit (chevauchement ≥ 2h30 avec la plage 21h30–06h30), l'agent doit avoir l'habilitation peutFaireNuit = true.
Bloquant
7. HABILITATION DÉPLACEMENT — Autorisation hors LPA
Si la JS entraîne un déplacement hors LPA, l'agent doit avoir l'habilitation peutEtreDeplace = true. S'applique uniquement en mode manuel (quand le système LPA ne peut pas calculer automatiquement).
Vigilance
8. GPT NUIT CONSÉCUTIVES — 2 GPT de nuit de suite
Il est interdit d'avoir 2 GPT de nuit consécutives. Une GPT est "de nuit" si ≥ 50% de ses journées contiennent la plage 00h00–04h00. Déclenche une VIGILANCE, pas une exclusion.
Vigilance
9. RÉGIME B/C — Durée minimum
Pour les agents en Régime B ou C, chaque JS doit avoir un TE minimum de 5h30 (330 min). Un poste trop court déclenche une VIGILANCE (score pénalisé) mais ne bloque pas l'affectation.
Bloquant
10. PRÉFIXE JS — Habilitations par code JS
Si des préfixes JS sont configurés pour un agent, le code de la JS imprévisible doit correspondre à l'un de ces préfixes. Ex. : si l'agent est habilité GIC, il peut prendre GIC004R mais pas PEY006R.

Règles de vigilance (non-bloquantes)

Vigilance
GPT_MIN — GPT trop courte
Une GPT doit normalement contenir minimum 3 JS. Si la JS simulée crée une GPT plus courte, c'est un point de vigilance (exception possible le dimanche).
Vigilance
GPT_MAX_AVANT_RP — GPT > 5 JS
Si la GPT dépasse 5 JS (mais reste ≤ 6), le prochain repos périodique devra être au moins un RP double (60h). Ce n'est pas une violation immédiate mais un avertissement pour la planification.
Vigilance
REPOS PÉRIODIQUES GPT — Avant et après
Le système vérifie que le repos périodique avant la GPT et après la GPT sont conformes aux durées requises (simple/double/triple selon la longueur de la GPT).

Déplacement LPA — Calcul automatique

Comment le système détermine si un agent est en déplacement et calcule ses horaires effectifs

Principe

Chaque agent a un LPA (Lieu Principal d'Affectation). Une JS peut être dans la LPA (pas de déplacement) ou hors LPA (déplacement avec temps de trajet). Le système calcule automatiquement le statut via la table LpaJsType.

Algorithme de calcul (computeEffectiveService)

1
Résoudre le JsType de la JS
Cherche d'abord par correspondance exacte (typeJs = JsType.code), puis par préfixe du codeJs.
2
Vérifier si la JS est dans la LPA de l'agent
Consulte la table LpaJsType : si le couple (LPA agent, JsType) existe → JS dans la LPA → pas de déplacement.
3
Vérifier les règles individuelles (AgentJsDeplacementRule)
Un override par agent peut forcer ou interdire le déplacement pour ce JsType précis, avec des temps de trajet spécifiques (aller/retour en minutes).
4
Calculer les horaires effectifs
heureDebutEffective = heureDebutJsType − tempsAller
heureFinEffective = heureFinJsType + tempsRetour
L'amplitude effective inclut ces temps de trajet.
💡 Exemple concret
Agent PANEL, LPA = Givors. JS PEY006R standard 21h00–05h00.
La JS PEY006R est hors LPA Givors → déplacement activé.
Règle de déplacement de PANEL : trajet aller = 30 min, retour = 1h10.
→ Horaires effectifs PANEL : 20h30–06h10 (amplitude = 9h40)
→ Horaires effectifs WERNER (si trajet aller = 60 min, retour = 0) : 20h00–05h00
⚠ Clé du bug corrigé
Quand le système cherche un remplaçant pour la JS de PANEL, il utilise les horaires standard du JsType (21h00–05h00) et non les horaires personnels de PANEL (20h30–06h10). Chaque candidat voit ensuite ses propres temps de trajet appliqués à ces horaires standards.

Import planning incrémental

Enrichir le planning consolidé — les imports sont cumulatifs, jamais destructifs

Ouvrir l'import planning

L'import enrichit le planning consolidé de manière incrémentale : chaque ligne est identifiée par son doublet (matricule, jour du planning). Si la ligne existe déjà, elle est mise à jour ; sinon elle est créée. Plusieurs imports peuvent coexister — on peut importer une équipe, puis une autre, sans effacer les données déjà présentes. Chaque import est tracé dans un journal (PlanningImport) mais ne "remplace" plus les précédents.

Formats acceptés

Excel (.xlsx / .xls)

Fichier tabulaire avec en-tête. Tolérant aux colonnes dans n'importe quel ordre.

📄 TXT tabulé

Fichier texte avec tabulations comme séparateurs. Export direct depuis certains systèmes RH.

Comportement

  • Les agents sont permanents : l'import ne supprime jamais un agent existant.
  • Un agent est identifié par son matricule (code immatriculation). Si l'agent existe déjà, son profil est mis à jour (upsert).
  • Chaque import est cumulatif : un même créneau (même agent + date + heure de début) est mis à jour si il existe déjà dans le planning.
  • En cas d'erreur sur une ligne, la ligne est ignorée (les autres sont importées).
  • Rétention automatique M-6 : les lignes de planning dont la date de fin est antérieure à 6 mois sont supprimées automatiquement après chaque import (nettoyage en arrière-plan, sans blocage). Les agents ne sont jamais supprimés.

Propositions d'habilitations post-import (admin)

Après chaque import, un panneau dédié visible uniquement par les admins propose des ajustements automatiques des habilitations (préfixes JS) des agents, calculés à partir de l'historique complet du planning. Le système ne modifie rien sans validation manuelle — il suggère, l'admin coche/décoche puis valide en un clic.

À ajouter — codes tenus non couverts

Pour chaque agent, le moteur liste les codeJs qu'il a réellement tenus (hors NPO, hors JS Z) et qui ne sont couverts par aucun de ses préfixes d'habilitation actuels. Chaque code non couvert est proposé tel quel (le préfixe le plus restrictif possible). Indique nb jours tenus et dernier jour.

À retirer — préfixes inutilisés

Inversement, pour chaque préfixe d'habilitation actuel, si aucun codeJs de l'historique ne le matche, il est proposé à la suppression. Symétrique de l'ajout, garde la liste d'habilitations alignée sur l'usage réel (un agent qui n'a fait que des JS Z « GIV Z » verra son préfixe GIV proposé au retrait).

Filtre cohérence JS Z
Les lignes JS sans charge réelle (suffixe « Z », préfixe FO, typeJs = DIS, préfixes admin ZeroLoadPrefix) ne comptent pas dans l'historique de référence — un agent placé sur une JS Z n'y a pas réellement « tenu » le code. Cela vaut pour les deux sens : pas d'ajout suggéré sur ces lignes, et un préfixe ne matchant que ces lignes est bien proposé au retrait.

Workflow de validation

1
Import lancé
Dès que l'import réussit, le panneau apparaît automatiquement sous le formulaire d'import.
2
Calcul à la demande
Recalculé à chaque ouverture (pas de cache) — reflète toujours l'historique courant. GET /api/habilitations/propositions.
3
Cases à cocher pré-cochées
Tous les ajouts ET retraits proposés sont sélectionnés par défaut. L'admin décoche ce qu'il refuse.
4
Validation en un clic
Bouton « Valider (N) » applique simultanément les ajouts ET les retraits. POST /api/habilitations/propositions/valider — rate-limité 10/min.
5
Audit
Action HABILITATION_AUTO_VALIDATED avec détails {prefixesAjoutes, prefixesRetires, habilitationsApres}. Idempotent : revalider ne crée pas de doublons.

Workflow de la simulation — Étapes détaillées

Ce que le moteur fait de A à Z pour chaque simulation

1
Pré-filtrage des agents

Avant même d'appliquer les règles, on exclut les agents inéligibles :

  • L'agent initialement affecté à la JS (on ne peut pas se remplacer soi-même)
  • Agents en absences inaptitude sur ce créneau
  • Agents déjà en service sur ce créneau (chevauchement horaire)
  • Agents dont les préfixes JS ne correspondent pas au code de la JS cible
2
Calcul du service effectif (LPA)
Pour chaque candidat, le système calcule ses horaires effectifs en tenant compte de son déplacement éventuel. Il part toujours des horaires standards du JsType (pas des horaires de l'agent absent), puis applique les temps de trajet propres au candidat.
3
Évaluation des 10 règles réglementaires
Pour chaque candidat, le moteur evaluerMobilisabilite() vérifie les 10 règles (amplitude, repos journalier, TE, GPT, etc.) et retourne un statut : CONFORME, VIGILANCE ou NON_CONFORME avec le détail de chaque règle.
4
Détection des conflits induits
Pour les candidats mobilisables, le système vérifie si ajouter cette JS ne crée pas de problèmes sur le planning futur de l'agent. Un conflit induit typique : l'agent serait en repos insuffisant avant sa prochaine JS planifiée.
5
Résolution en cascade (si nécessaire)
Si un candidat a un conflit induit résolvable (repos insuffisant avant une JS future), le système cherche récursivement un autre agent pour reprendre cette JS libératrice. Jusqu'à 10 niveaux de profondeur.
6
Construction des 4 scénarios parallèles
Le moteur génère systématiquement 4 scénarios en combinant deux axes : le périmètre du vivier (agents de réserve uniquement, ou tous les agents éligibles) et l'activation ou non du figeage. Chaque scénario est scoré globalement sur 0 à 100 et présenté côte à côte pour permettre la comparaison.
💡 Exemple complet
Trois JS imprévues sur la même journée (ex. 16 mars : GIC006R, PEY002S, VAL004N).
Le système analyse l'ensemble des agents éligibles puis produit 4 scénarios parallèles :

Réserve seule : couverture partielle avec les agents de réserve. Score global calculé sur la couverture totale.
Tous agents : ouverture à l'ensemble du vivier. Taux de couverture généralement supérieur.
Avec figeage : autorise à figer les agents déjà positionnés sur des JS en dernier recours pour libérer du monde.
Sans figeage : pas de remaniement des agents déjà planifiés. Cascade seule pour résoudre les conflits.

Règles de la cascade

  • Anti-cycle : un agent ne peut pas apparaître deux fois dans la même chaîne
  • Profondeur max : 10 niveaux (CASCADE_MAX_DEPTH = 10)
  • Condition de déclenchement : uniquement pour les conflits de type REPOS_INSUFFISANT

✓ Résolu

✗ Non résolvable

Aucun agent disponible pour couvrir une JS bloquante. L'impact est marqué comme BLOQUANT dans les résultats.

Figeage — DERNIER RECOURS

Libérer un agent en gelant une JS moins prioritaire

Quand tous les agents disponibles sont insuffisants, on peut libérer un agent planifié sur une JS DERNIER_RECOURS en "figeant" cette JS (elle restera non-couverte). Cet agent devient alors disponible pour l'imprévu urgent.

Fonctionnement

Sans figeage

Seuls les agents libres ou sur JS Z (suffixe Z, préfixe FO, typeJs = DIS, ou préfixes admin) sont candidats. Les agents sur JS planifiées restent sur leur poste.

Avec figeage

Les agents sur JS DERNIER_RECOURS peuvent être libérés. La JS figée reste sans couverture (pénalité réduite).

Indicateurs visuels

  • Badge JS figée — DERNIER RECOURS visible sur le scénario
  • L'onglet "Avec figeage" affiche le nombre d'agents supplémentaires disponibles
  • La fiche candidat affiche le champ jsSourceFigee avec la justification
⚠ Attention
Le figeage signifie qu'une JS DERNIER_RECOURS reste sans agent. L'encadrant doit décider si ce compromis est acceptable avant de valider le scénario.
ℹ Restriction importante
Seules les JS de type DERNIER_RECOURS peuvent être figées. Les JS OBLIGATOIRE ne peuvent jamais être libérées — elles restent prioritaires en toutes circonstances, y compris en simulation multi-JS.

Solveur unifié expérimental

Onglet additionnel sous flag — comparaison legacy ↔ nouveau moteur de cascade

⚗ Mode expérimental — non activé par défaut
Le solveur unifié est un nouveau moteur de cascade qui remplacera à terme la logique historique. Il est exposé dans un onglet dédié "Solveur unifié ⚗" quand la variable d'environnement FEATURE_UNIFIED_PRIMARY=1 est définie côté serveur. Le legacy reste la référence par défaut. Aucune modification de l'interface tant que le flag n'est pas activé.

Pourquoi un nouveau solveur ?

Le moteur historique sépare deux logiques distinctes :

Chaîne de remplacement
conflit horaire — un agent prend la JS imprévue, sa propre JS source est libérée et reprise par un autre agent.
Cascade conflits induits
conflit RH aval — repos insuffisant ou GPT/TE qui dépassent leurs limites après avoir pris l'imprévu.

Sur le terrain, ces deux mécanismes se combinent souvent dans une même résolution. Le solveur unifié les fusionne dans une boucle récursive unique : peu importe la nature du conflit, toute conséquence d'une affectation devient un nouveau besoin résolu de la même manière.

Lecture de l'onglet "Solveur unifié ⚗"

1
Bandeau expérimental + agrégat de comparaison
En tête, un encadré rappelle le statut expérimental. Juste en dessous, un agrégat compact compare au legacy : X match N1 ↔ legacy (= mêmes agents retenus), unified seul (JS couvertes par le nouveau moteur que le legacy laisse non couvertes), legacy seul (l'inverse), et le budget consommé.
2
Recommandations par JS
Pour chaque JS sélectionnée, les solutions classées par score métier décroissant. Chacune affiche un agent N1 (qui prend l'imprévu), une profondeur de chaîne, et un niveau de risque sous forme de badge.
3
Alternatives avancées (repliées par défaut)
Si une JS expose des solutions "déconseillées" (chaînes lourdes, agents proches de la saturation), elles sont disponibles dans une zone repliable séparée — jamais affichées comme recommandation principale. Le décideur peut les déplier pour les consulter mais elles ne sont pas suggérées.
4
Séquence terrain testée
Si un test de séquence forcée est configuré (ex: "Chennouf → Brouillat → Leguay"), un encadré bleu en bas affiche son verdict POSSIBLE ou IMPOSSIBLE avec la trace détaillée de chaque étape (faisabilité, conséquences, raison d'échec exacte).

Niveaux de risque

NiveauCouleurSignification
OKVertToutes les affectations respectent les règles RH avec marge confortable. Exploitable sans réserve.
VigilanceAmbreAu moins une affectation en vigilance (statut RH dégradé mais conforme). À signaler à l'opérateur, mais utilisable.
DéconseilléeOrangeAu moins un agent dans la chaîne a un score métier ≤ 30 (saturé GPT, marge de repos faible…). Solution exposée mais non recommandée — repliée par défaut.

Activation et rollback

🔧 Procédure côté serveur
Activation : 
echo "FEATURE_UNIFIED_PRIMARY=1" >> .env
pm2 restart pointrh
Rollback immédiat : 
sed -i 's/^FEATURE_UNIFIED_PRIMARY=1/#FEATURE_UNIFIED_PRIMARY=1/' .env
pm2 restart pointrh
L'onglet disparaît instantanément, le legacy reprend strictement comme avant.
🔍 Périmètre d'affichage
L'onglet n'apparaît que sur les scénarios cascade (Tous agents — Cascade et Tous agents + Cascade + Figeage). Sur les scénarios sans logique cascade, l'analyse comparative serait sans intérêt et l'onglet n'est pas généré. Si vous changez de scénario en cours d'exploration et que l'onglet disparaît, vous retombez automatiquement sur le Résumé.

À surveiller pendant l'activation progressive

  • Logs serveur : pm2 logs pointrh | grep UNIFIED — agrégats par scénario, budgets consommés, séquences cibles trouvées
  • Cohérence legacy ↔ unified : si nbN1Match est faible (< 50% des JS), la divergence mérite investigation
  • Solutions DECONSEILLEE remontées comme pertinentes par les utilisateurs — le seuil est paramétrable (constante SEUIL_SCORE_DECONSEILLE)
  • Charge : si budgetTotal dépasse régulièrement 5000, augmenter CASCADE_EVAL_BUDGET ou réduire maxSolutionsParJs

Procédure de déploiement en production

✓ Pas de migration DB
Le solveur unifié n'introduit ni nouveau schéma Prisma ni nouvelle table. La sortie API reste rétrocompatible — les nouveaux champs (unifiedReport, scenarioReserveOnlyCascade*) sont optionnels. Un déploiement de code seul, flags désactivés, ne change rien au comportement perçu par l'utilisateur.

Phase 1 — Déploiement code (zéro risque)

Sur le VPS, lancer le script standard :

cd /var/www/Point-RH
./deploy-rh.sh

Build + restart PM2. Test de fumée recommandé : ouvrir une simulation multi-JS connue et vérifier que 8 scénarios apparaissent (au lieu de 6 — les deux nouveaux scénarios Réserve × Cascade). Aucun onglet "Solveur unifié ⚗" ne doit être visible à ce stade.

Phase 2 — Activation shadow (logs serveur uniquement)

./toggle-unified.sh shadow on

Active UNIFIED_SHADOW=1 et redémarre PM2. Le solveur tourne en parallèle de l'allocator legacy, écrit ses agrégats dans la console serveur. Aucun impact UI. À surveiller pendant 24-48h via pm2 logs pointrh | grep UNIFIED.

Phase 3 — Bascule UI (visible utilisateurs)

./toggle-unified.sh ui on

Active FEATURE_UNIFIED_PRIMARY=1. L'onglet "Solveur unifié ⚗" apparaît sur les scénarios cascade (multi-JS) et sur le panneau d'analyse d'imprévu (simulation simple). Le legacy reste mode par défaut sur tous les autres onglets.

↩ Rollback immédiat (à tout moment)
Une seule commande désactive l'UI, en gardant le shadow pour audit : 
./toggle-unified.sh ui off
Pour un rollback complet (UI + shadow), utiliser ./toggle-unified.sh full off. Le restart PM2 est automatique. L'onglet disparaît instantanément.

Vérification de l'état des flags

./toggle-unified.sh status

Affiche l'état actuel des deux flags sans rien modifier.

CommandeEffet
./toggle-unified.sh statusLecture seule — affiche les flags actifs
./toggle-unified.sh shadow on / offActive / désactive les logs serveur
./toggle-unified.sh ui on / offActive / désactive l'onglet UI

Mode "thorough" (analyse approfondie ponctuelle)

Par défaut, le solveur unifié tourne en mode léger (5 solutions par JS, budget 3000, sans phase exhaustive) pour ne pas peser plus de quelques secondes par simulation cascade. Pour activer le mode complet (12 solutions, budget 12000, exhaustif activé, diagnostics nominatifs Chennouf-Brouillat-Leguay et test de séquence forcée), ajouter la variable :

echo "UNIFIED_THOROUGH=1" >> /var/www/Point-RH/.env
pm2 restart pointrh --update-env

Réservé aux situations d'investigation — ce mode peut prendre 1-3 minutes par simulation selon la taille du dataset, contre quelques secondes en mode léger. Sur la prod permanente, laisser cette variable absente.

Supprimer un agent — Soft delete

ℹ️ Suppression logique uniquement
La suppression dans Point RH est toujours logique (soft delete). L'agent n'apparaît plus dans les listes et ne peut plus être sélectionné dans les simulations, mais ses données restent en base pour assurer la traçabilité historique. Seuls les administrateurs peuvent supprimer. L'action est tracée avec l'identité de l'admin et la date.

Un agent supprimé peut être restauré à tout moment depuis l'écran /admin/agents-supprimes (voir §Administration).

Points de vigilance

Points de vigilance importants
  • Les données agents sont importées depuis le planning Excel/TXT. Un fichier comportant des erreurs (matricule manquant, champs vides) peut provoquer des agents mal créés.
  • Les habilitations JS et autorisations (nuit, déplacement) doivent être vérifiées après chaque import — le fichier source ne les contient pas toujours.
  • Un agent sans habilitation JS ne sera jamais proposé comme candidat, même s'il est disponible.
  • Sans LPA configuré, le calcul de déplacement devient « indéterminable » et l'application bascule en mode manuel.

Impact concret des paramètres sur les calculs

Ce qui se passe quand on modifie une valeur — exemples avant/après

Prise d'effet
Toute modification prend effet à la simulation suivante, sans redémarrage. Les simulations archivées ne sont pas recalculées.

7 scénarios avant/après — parcourez les onglets

Relever amplitude.general de 11h à 11h30

Avant : agent affecté 05h00–16h15 (11h15) → NON CONFORME (dépassement 15 min).
Après : CONFORME (marge 15 min). Le score passe de 0 à ~95.
Risque : dépassement des accords collectifs CTF. À ne faire qu'avec validation écrite du service RH.

Réduire reposJournalier.apresNuit de 14h à 12h

Avant : fin de poste nuit 06h00 → reprise min 20h00.
Après : reprise dès 18h00 — 8+ candidats additionnels CONFORMES pour les postes de soirée.
Risque : ce paramètre protège d'un cumul de fatigue après nuit. Le réduire est un risque sanitaire et réglementaire majeur.

Passer gpt.max de 6 à 7 jours

Avant : agent à 6 JS ne peut pas être affecté à un 7ᵉ.
Après : 7ᵉ JS autorisé. Il faut aussi relever gpt.maxAvantRP (sinon validation de cohérence refusée).
Effet collatéral : l'alerte VIGILANCE du 5ᵉ JS se décale d'un jour.

Modifier periodeNocturne.seuilJsNuit de 2h30 à 1h30

Avant : poste 18h00–22h00 (1h30 chevauchement) → NON considéré nuit.
Après : CLASSÉ NUIT. Agents non habilités nuit bloqués. Amplitude max = 11h/10h30 réserve. Repos post = 14h.
Impact : peut bloquer des dizaines d'affectations jusqu'alors autorisées.

Désactiver le code NPO exclu MA

Avant : agent malade (code MA) exclu automatiquement.
Après : agent considéré, évalué sur règles standard, peut apparaître CONFORME — indésirable en pratique.
Recommandation : préférer retirer le code NPO d'un agent spécifique côté planning plutôt que désactiver globalement.

Associer le type de JS NPE à un LPA supplémentaire (LPA1)

Avant : agents de LPA1 en déplacement sur NPE (amplitude + trajets).
Après : plus de déplacement sur NPE pour LPA1. Nombreux agents précédemment NON CONFORMES (AMPLITUDE) deviennent CONFORMES.
Note : les règles de trajet individuelles restent en base mais inactives tant que NPE est dans le LPA.

Passer le type de JS FIX en DERNIER_RECOURS

Avant : FIX OBLIGATOIRE. Pénalité forte si non couvert, agent titulaire figé impossible.
Après : pénalité réduite. Le moteur peut libérer l'agent titulaire FIX pour servir un GIV plus critique.
Usage : réserver aux postes non opérationnellement critiques (formations, réunions internes).

Cybersécurité — Authentification

JWT, cookies HTTP-only, bcrypt, durée de session

Point RH traite des données personnelles sensibles (identité agents, plannings, données RH). L'authentification est un maillon critique.

Mécanisme JWT

Jeton signé
À la connexion, un JWT (JSON Web Token) est généré et signé avec JWT_SECRET. Contient l'id utilisateur, le rôle, et la date d'expiration.
Cookie HTTP-only
Stocké dans un cookie invisible au JavaScript — protège contre le vol de session par XSS.
Expiration 8 heures
Une reconnexion est demandée après ce délai. Le cookie est automatiquement invalidé.
bcrypt 12 rounds
Les mots de passe sont chiffrés avec bcrypt (12 rounds de coût). Impossible à retrouver à partir du haché en base.
🔑 JWT_SECRET en production
La valeur par défaut de développement DOIT être remplacée en production par une chaîne aléatoire de 32+ caractères. Stockée dans .env, jamais commitée dans git.

Protection des données personnelles (RGPD)

Obligations et principes appliqués

Catégories de données traitées

Identité
Nom, prénom, matricule, adresse e-mail professionnelle.
Données professionnelles
Plannings, journées de service, habilitations, grades, postes d'affectation.

Principes appliqués

✓ Conformité RGPD
  • Minimisation — seules les données strictement nécessaires à la gestion RH sont collectées.
  • Conservation limitée — les données archivées (agents supprimés) sont conservées uniquement à des fins de traçabilité.
  • Accès restreint — authentification obligatoire, rôles USER/ADMIN, journal d'audit.
  • Sécurité — HTTPS, cookies HTTP-only, mots de passe bcrypt, headers de sécurité HTTP (CSP, HSTS, X-Frame-Options).
Droit d'accès et de rectification
Tout agent référencé dans Point RH peut demander (via son manager ou le service RH) la consultation et la rectification de ses données personnelles.

Sécurité des accès réseau

HTTPS, CSP, HSTS, rate limiting

Risques principaux et bonnes pratiques

Ce qu'il faut connaître et comment s'en protéger

Les 4 risques principaux

Risque 1 — Fuite de données personnelles
Un export (liste d'agents, résultats de simulation) transmis hors des canaux sécurisés (e-mail perso, clé USB non chiffrée) constitue une violation RGPD.
Risque 2 — Modification non autorisée
Un compte compromis (mot de passe volé ou trop simple) permettrait à un tiers de modifier des habilitations, règles métier, ou d'importer un planning falsifié.
Risque 3 — Mauvaise interprétation des résultats
Un statut VIGILANCE ne signifie pas « refusé ». Un utilisateur inattentif peut affecter un agent en situation limite sans vérifier — créant un risque réglementaire réel.
Risque 4 — Données de planning obsolètes
Si l'import n'est pas à jour, les calculs de GPT et de repos seront incorrects. L'application peut proposer des agents dont la situation réelle diffère.

Bonnes pratiques utilisateurs

✓ Pour tous les utilisateurs
  • Choisir un mot de passe long et unique (12 caractères minimum, mélange lettres/chiffres/symboles).
  • Ne jamais partager son mot de passe — pas même à un collègue ou à son responsable.
  • Se déconnecter de Point RH dès la fin de l'utilisation.
  • Verrouiller son poste en cas d'absence (Windows+L).
  • Ne pas stocker d'exports sur supports non chiffrés (clés USB standards, e-mail perso).
  • Signaler immédiatement toute activité suspecte sur son compte à l'administrateur.
  • Vérifier systématiquement le détail d'un agent VIGILANCE avant de l'affecter.
  • S'assurer que le planning importé est la dernière version officielle avant toute simulation.

Bonnes pratiques administrateurs

✓ Pour les administrateurs
  • Réviser régulièrement la liste des comptes actifs — désactiver les inutilisés.
  • N'accorder le rôle ADMIN qu'aux personnes qui en ont réellement besoin.
  • Contrôler le journal d'audit régulièrement (actions inhabituelles, modifications de règles).
  • Sauvegarder la base de données avant toute opération de maintenance ou mise à jour.
  • Changer le JWT_SECRET en production — ne jamais laisser la valeur par défaut.
  • Tester les règles métier en préproduction avant de les modifier en production.
  • Documenter chaque modification des règles (qui, quand, pourquoi, avant/après).
  • Former les nouveaux utilisateurs avant de leur donner accès.
  • Contrôler les exports de paramétrage — ils contiennent toute la configuration métier.
