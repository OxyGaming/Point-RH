Présentation de Point RH

Système de simulation de remplacement pour les journées de service SNCF

Point RH est un outil d'aide à la décision permettant aux encadrants de trouver rapidement quel agent peut remplacer un collègue absent sur une Journée de Service (JS) imprévisible, en vérifiant automatiquement toutes les règles réglementaires (repos, amplitude, GPT, TE…).

🔍 Simulation JS Simple

Un agent est absent sur une JS précise. Le système identifie qui peut le remplacer, en cascade si nécessaire.

Mode principal

📅 Simulation Multi-JS

Plusieurs JS sont à couvrir simultanément. Le système propose des répartitions globales optimisées sur plusieurs scénarios.

Mode avancé

👤 Fiches agents

Profil complet de chaque agent : habilitations, LPA, régime, règles de déplacement personnalisées.

📥 Import planning

Chargement des plannings depuis un fichier Excel/TXT exporté du système RH source.

Glossaire — Définitions des termes

Tous les termes utilisés dans Point RH, avec leur définition précise

Termes fondamentaux

JS
Journée de Service
Un poste de travail planifié, défini par une date, une heure de début, une heure de fin, un code (ex. PEY006R) et un type. C'est l'unité de base du planning.
Ex : JS du 17 mars, 21h00–05h00, code GIC004R
NPO
Non-Présence Organisée
Toute absence planifiée : congés, repos hebdomadaire (RU), repos périodique (RP), maladie, inaptitude. Ces événements figurent dans le planning mais ne sont pas des journées de service.
Ex : Congé du 20 au 22 mars, repos périodique le 18 mars
GPT
Grande Période de Travail
Séquence de journées de service consécutives entre deux Repos Périodiques (RP). Maximum 6 JS par GPT. Détermine le cumul TE, la nature nuit/jour, et déclenche les vérifications de repos périodiques.
Ex : Lundi JS, Mardi JS, Mercredi JS, Jeudi RP → GPT de 3 jours
TE
Travail Effectif
Durée réelle de travail d'une journée de service, hors temps de trajet. Limite par JS : 10h standard, 8h30 de nuit, 12h en remplacement. Cumul maximum sur une GPT : 48h.
JS de 21h00 à 05h00 = amplitude 8h, TE = 7h30 (si 30min de trajet)
RP
Repos Périodique
Période de repos réglementaire qui clôture une GPT. Trois niveaux : simple (36h), double (60h), triple (84h). Une GPT > 5 JS doit être suivie d'un RP double minimum.
RP simple : partir jeudi soir, reprendre samedi matin (36h gap)
LPA
Lieu de Prise d'Attachement
La base géographique de référence d'un agent. Si une JS se déroule hors LPA, l'agent est en déplacement et des temps de trajet s'ajoutent à son amplitude effective.
LPA = Givors. JS à Lyon Part-Dieu → hors LPA → déplacement
Amplitude
Amplitude de service
Durée totale d'une journée de service, du début effectif à la fin effective. Inclut les temps de trajet si l'agent est en déplacement. Limite : 11h standard, jusqu'à 13h en déplacement + remplacement.
JS 20h00–06h10 = amplitude 10h10. Avec trajet +1h = amplitude effective 11h10
Repos journalier
Repos entre deux postes
Durée minimale obligatoire entre la fin d'un poste et le début du suivant. Minimum 12h standard, 14h après un poste de nuit, 10h pour un agent de réserve (1 fois/GPT).
Fin JS le 17 mars à 06h10. Prochain poste le 17 mars à 20h00 → repos = 13h50 ✓

Termes de la simulation

Conforme
Toutes les règles réglementaires sont respectées. L'agent est mobilisable sans réserve. Badge vert dans l'interface.
Vigilance
Une règle non-bloquante est proche de son seuil ou légèrement dépassée. L'agent peut être mobilisé avec réserve. L'encadrant doit valider manuellement. Badge jaune.
Non-conforme
Une ou plusieurs violations bloquantes existent. L'agent ne peut pas prendre ce poste. Badge rouge.
Cascade
Mécanisme de résolution en chaîne. Si l'agent A est bloqué par un conflit repos (sa JS B l'en empêche), on cherche un agent C pour reprendre la JS B, libérant ainsi A. Jusqu'à 10 niveaux.
A doit faire JS1 mais bloqué par JS2 → C reprend JS2 → A libre pour JS1
Figeage
Action de "geler" une JS de type DERNIER_RECOURS pour libérer l'agent planifié dessus, afin qu'il puisse prendre un imprévu plus urgent. La JS figée reste non-couverte.
JS Z / JS de type Z
Journée de service sans charge réelle (placeholder). Un agent prévu sur une JS Z peut être réaffecté sans avoir besoin de trouver un remplaçant pour cette JS d'origine. Bonus +15 pts au score.
JS FO
Journée de service de type Formation/Organisation. Identifiée par le préfixe "FO". Peut être réaffectée comme une JS Z si l'agent est mobilisé sur un imprévu.
JsType
Référentiel central d'un type de journée de service. Définit les horaires standards (indépendants des trajets), la durée, si c'est une JS de nuit, le régime, et la flexibilité (OBLIGATOIRE ou DERNIER_RECOURS).
JsType "PEY006R" : 21h00–05h00, 8h, estNuit=true, OBLIGATOIRE
OBLIGATOIRE
Flexibilité d'un JsType. La JS doit être couverte. Pénalité maximale si non-couverte en simulation multi-JS.
DERNIER_RECOURS
Flexibilité d'un JsType. La JS peut rester non-couverte avec une pénalité réduite. L'agent planifié peut être libéré par figeage pour couvrir une JS OBLIGATOIRE plus urgente.
Déplacement
Situation où une JS se déroule hors du LPA de l'agent. Des temps de trajet (aller/retour) s'ajoutent automatiquement à l'amplitude. Nécessite l'habilitation peutEtreDeplace.
Régime B / Régime C
Régimes réglementaires spéciaux imposant un Travail Effectif minimum de 5h30 par journée de service. Les agents en Régime B ou C ne peuvent pas prendre de postes trop courts.
Agent de réserve
Agent dédié aux remplacements. Bénéficie de règles légèrement assouplies : repos minimum 10h (au lieu de 12h) une fois par GPT, amplitude nuit réduite à 10h30. Bonus +10 pts au score.

Règles réglementaires

Les 10 règles vérifiées automatiquement par le moteur de simulation

⚠ Règles bloquantes vs. vigilance
Une règle BLOQUANTE empêche la mobilisation. Une règle VIGILANCE avertit mais permet la mobilisation avec validation encadrant.
Bloquant
1. AMPLITUDE — Durée totale de la journée
L'amplitude (durée totale du début à la fin, trajet inclus) ne doit pas dépasser le seuil autorisé. La limite varie selon le contexte :
Standard : 11h (660 min) Nuit : 11h (660 min) Réserve + nuit : 10h30 (630 min) Déplacement : 12h (720 min) Déplacement + remplacement : 13h (780 min)
Bloquant
2. REPOS JOURNALIER — Repos entre deux postes
La durée entre la fin du dernier poste et le début de l'imprévu doit respecter un minimum. Varie selon le contexte du dernier poste :
Standard : 12h (720 min) Après nuit : 14h (840 min) Réserve + remplacement (1×/GPT) : 10h (600 min)
⚠ Si TE > 6h sans coupure interne, ajouter +20 min au repos minimum requis.
Bloquant
3. TRAVAIL EFFECTIF — Durée de travail par JS
Le travail effectif (hors trajet) par journée ne peut pas dépasser :
Standard : 10h (600 min) Nuit : 8h30 (510 min) Remplacement : 12h (720 min)
Bloquant
4. TE GPT 48H — Cumul travail effectif sur la GPT
Le cumul des TE de toutes les JS dans la GPT en cours plus la JS imprévisible ne doit pas dépasser 48 heures (2880 min).
💡 Exemple
GPT en cours : JS1=8h + JS2=8h30 + JS3=9h = 25h30 cumulés.
Imprévu = 8h. Total = 33h30 ✓
Si GPT avait déjà 45h cumulés + imprévu 5h = 50h → VIOLATION
Bloquant
5. GPT_MAX — Longueur maximale de la GPT
Une GPT ne peut pas dépasser 6 journées de service consécutives. Le système injecte la JS simulée dans le planning pour calculer la GPT résultante.
💡 Exemple
Agent en GPT de 5 JS. Ajouter 1 JS → GPT = 6 ✓
Agent en GPT de 6 JS. Ajouter 1 JS → GPT = 7 → VIOLATION
Bloquant
6. HABILITATION NUIT — Autorisation postes de nuit
Si la JS est une JS de nuit (chevauchement ≥ 2h30 avec la plage 21h30–06h30), l'agent doit avoir l'habilitation peutFaireNuit = true.
Bloquant
7. HABILITATION DÉPLACEMENT — Autorisation hors LPA
Si la JS entraîne un déplacement hors LPA, l'agent doit avoir l'habilitation peutEtreDeplace = true. S'applique uniquement en mode manuel (quand le système LPA ne peut pas calculer automatiquement).
Bloquant
8. GPT NUIT CONSÉCUTIVES — 2 GPT de nuit de suite
Il est interdit d'avoir 2 GPT de nuit consécutives. Une GPT est "de nuit" si ≥ 50% de ses journées contiennent la plage 00h00–04h00.
Bloquant
9. RÉGIME B/C — Durée minimum
Pour les agents en Régime B ou C, chaque JS doit avoir un TE minimum de 5h30 (330 min). Un poste trop court est refusé.
Bloquant
10. PRÉFIXE JS — Habilitations par code JS
Si des préfixes JS sont configurés pour un agent, le code de la JS imprévisible doit correspondre à l'un de ces préfixes. Ex. : si l'agent est habilité GIC, il peut prendre GIC004R mais pas PEY006R.

Règles de vigilance (non-bloquantes)

Vigilance
GPT_MIN — GPT trop courte
Une GPT doit normalement contenir minimum 3 JS. Si la JS simulée crée une GPT plus courte, c'est un point de vigilance (exception possible le dimanche).
Vigilance
GPT_MAX_AVANT_RP — GPT > 5 JS
Si la GPT dépasse 5 JS (mais reste ≤ 6), le prochain repos périodique devra être au moins un RP double (60h). Ce n'est pas une violation immédiate mais un avertissement pour la planification.
Vigilance
REPOS PÉRIODIQUES GPT — Avant et après
Le système vérifie que le repos périodique avant la GPT et après la GPT sont conformes aux durées requises (simple/double/triple selon la longueur de la GPT).

Déplacement LPA — Calcul automatique

Comment le système détermine si un agent est en déplacement et calcule ses horaires effectifs

Principe

Chaque agent a un LPA (Lieu de Prise d'Attachement). Une JS peut être dans la LPA (pas de déplacement) ou hors LPA (déplacement avec temps de trajet). Le système calcule automatiquement le statut via la table LpaJsType.

Algorithme de calcul (computeEffectiveService)

1
Résoudre le JsType de la JS
Cherche d'abord par correspondance exacte (typeJs = JsType.code), puis par préfixe du codeJs.
2
Vérifier si la JS est dans la LPA de l'agent
Consulte la table LpaJsType : si le couple (LPA agent, JsType) existe → JS dans la LPA → pas de déplacement.
3
Vérifier les règles individuelles (AgentJsDeplacementRule)
Un override par agent peut forcer ou interdire le déplacement pour ce JsType précis, avec des temps de trajet spécifiques (aller/retour en minutes).
4
Calculer les horaires effectifs
heureDebutEffective = heureDebutJsType − tempsAller
heureFinEffective = heureFinJsType + tempsRetour
L'amplitude effective inclut ces temps de trajet.
💡 Exemple concret
Agent PANEL, LPA = Givors. JS PEY006R standard 21h00–05h00.
La JS PEY006R est hors LPA Givors → déplacement activé.
Règle de déplacement de PANEL : trajet aller = 30 min, retour = 1h10.
→ Horaires effectifs PANEL : 20h30–06h10 (amplitude = 9h40)
→ Horaires effectifs WERNER (si trajet aller = 60 min, retour = 0) : 20h00–05h00
⚠ Clé du bug corrigé
Quand le système cherche un remplaçant pour la JS de PANEL, il utilise les horaires standard du JsType (21h00–05h00) et non les horaires personnels de PANEL (20h30–06h10). Chaque candidat voit ensuite ses propres temps de trajet appliqués à ces horaires standards.

Import planning

Chargement des données de planning depuis le système RH source

L'import charge les plannings de tous les agents pour une période donnée. Un import est lié à un importId unique. Seul le dernier import est actif. Les simulations utilisent toujours l'import actif.

Formats acceptés

📊 Excel (.xlsx / .xls)

Fichier tabulaire avec en-tête. Tolérant aux colonnes dans n'importe quel ordre.

📄 TXT tabulé

Fichier texte avec tabulations comme séparateurs. Export direct depuis certains systèmes RH.

Colonnes attendues (24 champs)

GroupeColonnes
AgentUCH, CODE UCH, NOM, PRENOM, CODE IMMATRICULATION, CODE APES, GRADE
PlanningDATE DEBUT POP, HEURE DEBUT POP, HEURE FIN POP, DATE FIN POP
DuréesAMPLITUDE (100e/h), DUREE EFFECTIVE (HH:MM)
TypeJS/NPO, CODE JS/NPO, TYPE JS/FAM NPO, VALEUR NPO
RoulementUCH JS, CODE ROULEMENT JS, NUMERO JS

Comportement

  • Les agents sont permanents : l'import ne supprime jamais un agent existant.
  • Un agent est identifié par son matricule (code immatriculation). Si l'agent existe déjà, son profil est mis à jour (upsert).
  • L'import précédent est désactivé, le nouveau devient l'import actif.
  • En cas d'erreur sur une ligne, la ligne est ignorée (les autres sont importées).

Simulation JS Simple — Vue d'ensemble

Trouver le remplaçant idéal pour une journée de service imprévisible

La simulation simple est déclenchée depuis la fiche agent : on clique sur la JS de l'agent absent. Le système analyse automatiquement tous les autres agents et retourne deux groupes de résultats en parallèle :

  • Sans figeage : agents disponibles sans modifier aucune JS planifiée
  • Avec figeage : inclut les agents qu'on peut libérer en figeant une JS DERNIER_RECOURS

Les 4 onglets de résultats

✅ Mobilisables

Agents CONFORME — toutes les règles sont respectées, aucun conflit induit. Mobilisation sans réserve.

⚠ Vigilance

Agents VIGILANCE — mobilisables mais avec un point d'attention. Validation encadrant requise.

✗ Non utilisables

Agents REFUSÉ — violations bloquantes, ils ne peuvent pas prendre ce poste.

🔗 Scénarios

Propositions de réorganisation en cascade. Un agent bloqué peut être libéré par réaffectation d'un autre.

🔄

Workflow de la simulation — Étapes détaillées

Ce que le moteur fait de A à Z pour chaque simulation

1
Pré-filtrage des agents

Avant même d'appliquer les règles, on exclut les agents inéligibles :

  • L'agent initialement affecté à la JS (on ne peut pas se remplacer soi-même)
  • Agents en absences inaptitude sur ce créneau
  • Agents déjà en service sur ce créneau (chevauchement horaire)
  • Agents dont les préfixes JS ne correspondent pas au code de la JS cible
2
Calcul du service effectif (LPA)
Pour chaque candidat, le système calcule ses horaires effectifs en tenant compte de son déplacement éventuel. Il part toujours des horaires standards du JsType (pas des horaires de l'agent absent), puis applique les temps de trajet propres au candidat.
3
Évaluation des 10 règles réglementaires
Pour chaque candidat, le moteur evaluerMobilisabilite() vérifie les 10 règles (amplitude, repos journalier, TE, GPT, etc.) et retourne un statut : CONFORME, VIGILANCE ou NON_CONFORME avec le détail de chaque règle.
4
Détection des conflits induits
Pour les candidats mobilisables, le système vérifie si ajouter cette JS ne crée pas de problèmes sur le planning futur de l'agent. Un conflit induit typique : l'agent serait en repos insuffisant avant sa prochaine JS planifiée.
5
Résolution en cascade (si nécessaire)
Si un candidat a un conflit induit résolvable (repos insuffisant avant une JS future), le système cherche récursivement un autre agent pour reprendre cette JS libératrice. Jusqu'à 10 niveaux de profondeur.
6
Construction des scénarios
Chaque solution cascade devient un scénario, scoré de 0 à 100. Les scénarios sont triés par score décroissant (maximum 5 propositions).
💡 Exemple complet
BELABBAS est absent le 16 mars (JS GIC006R 20h30–04h30).
Le système analyse 23 agents → 4 mobilisables directs, 1 vigilance, 19 refusés, 2 scénarios.

Scénario 1 : SEFOUHI reprend directement. Toutes règles OK. Score 92/100.
Scénario 2 : PANEL peut reprendre MAIS sa prochaine JS (17 mars PEY006R) crée un conflit repos. → WERNER reprend la JS du 17 mars de PANEL → PANEL libéré pour le 16 mars. Score 54/100.
🔗

Résolution en cascade

Comment le système libère un agent bloqué par un conflit de repos

La cascade est déclenchée quand un candidat est bloqué par un conflit de repos journalier : il pourrait prendre l'imprévu si et seulement si sa prochaine JS planifiée était reprise par quelqu'un d'autre. Le système cherche récursivement ce "quelqu'un d'autre".

Règles de la cascade

  • Anti-cycle : un agent ne peut pas apparaître deux fois dans la même chaîne
  • Profondeur max : 10 niveaux (CASCADE_MAX_DEPTH = 10)
  • Condition de déclenchement : uniquement pour les conflits de type REPOS_INSUFFISANT
  • Horaires de référence : toujours les horaires standards du JsType (pas les horaires personnels de l'agent bloqué)
🔮 Cascade profonde (niv. 2+)
Si WERNER lui-même est bloqué par un conflit repos sur sa prochaine JS, le système cherche un agent D pour libérer WERNER, qui libère PANEL, qui prend l'imprévu. Et ainsi de suite jusqu'à 10 niveaux.

Résultats de la cascade

✓ Résolu

Un remplaçant conforme a été trouvé pour chaque JS bloquante. La chaîne est complète. Le scénario est proposé.

✗ Non résolvable

Aucun agent disponible pour couvrir une JS bloquante. L'impact est marqué comme BLOQUANT dans les résultats.

Scénarios de réorganisation

Comment lire et interpréter les scénarios proposés

Structure d'un scénario

  • Conformité finale : CONFORME VIGILANCE NON_CONFORME
  • Score : 0 à 100 (plus c'est haut, mieux c'est)
  • Modifications : liste des changements à opérer (qui reprend quoi)
  • Profondeur cascade : nombre de niveaux de réorganisation
  • Impacts en cascade : avertissements sur les agents de la chaîne

Lecture de la chaîne de réorganisation

Chaque nœud de la chaîne affiche pour chaque agent :

  • PREND : la JS qu'il va reprendre (horaires JsType standard → horaires effectifs si déplacement)
  • LIBÈRE : la JS qu'il cède pour que le niveau suivant puisse se libérer
  • Le motif de vigilance si l'agent est en statut jaune
  • Le Détail des règles (repos journalier, amplitude, TE GPT, GPT, RP périodiques) — cliquable pour voir tous les calculs
✓ Conseil de lecture
Un scénario avec conformité CONFORME + 1 modification (solution directe) est idéal. Plus le nombre de modifications et la profondeur augmentent, plus la réorganisation est complexe et risquée.

Le score mesure la qualité de la mobilisation d'un agent. Plus il est élevé, plus l'agent est un bon choix : règles respectées, planning peu chargé, agent de réserve, peu de conflits induits.

Figeage — DERNIER RECOURS

Libérer un agent en gelant une JS moins prioritaire

Quand tous les agents disponibles sont insuffisants, on peut libérer un agent planifié sur une JS DERNIER_RECOURS en "figeant" cette JS (elle restera non-couverte). Cet agent devient alors disponible pour l'imprévu urgent.

Fonctionnement

Sans figeage

Seuls les agents libres ou sur JS Z sont candidats. Les agents sur JS planifiées restent sur leur poste.

Avec figeage

Les agents sur JS DERNIER_RECOURS peuvent être libérés. La JS figée reste sans couverture (pénalité réduite).

Indicateurs visuels

  • Badge 🔒 JS figée — DERNIER RECOURS visible sur le scénario
  • L'onglet "Avec figeage" affiche le nombre d'agents supplémentaires disponibles
  • La fiche candidat affiche le champ jsSourceFigee avec la justification
⚠ Attention
Le figeage signifie qu'une JS DERNIER_RECOURS reste sans agent. L'encadrant doit décider si ce compromis est acceptable avant de valider le scénario.

Simulation Multi-JS

Couvrir plusieurs JS simultanément avec une vision globale

La simulation multi-JS permet de sélectionner plusieurs journées de service à couvrir en même temps (plusieurs absents, ou un événement exceptionnel). Le système propose 4 scénarios en parallèle selon deux axes :

Axe 1 — Périmètre agents :
Réserve uniquement ou Tous les agents
Axe 2 — Figeage :
Sans figeage ou Avec figeage

Les 4 scénarios générés

#PérimètreFigeageUsage
1Réserve seuleNonIdéal — minimise l'impact
2Réserve seuleOuiRéserve + libérations
3Tous agentsNonCouverture maximale
4Tous agentsOuiCouverture maximale + libérations

Workflow multi-JS

1
Sélection des JS à couvrir
L'encadrant coche les JS à couvrir dans la liste des JS de l'import actif. Les JS sélectionnées deviennent les "cibles".
2
Calcul du service effectif (LPA)
Pour chaque couple (agent, JS cible), le système calcule les horaires effectifs basés sur les horaires standards du JsType (pas les horaires personnels de l'agent absent). Chaque candidat voit ses propres trajets appliqués.
3
Évaluation des candidats par JS
Pour chaque JS cible, le système évalue tous les agents éligibles et attribue un score de pertinence. Même logique que la simulation simple.
4
Affectation optimale (greedy)
Le meilleur candidat (score le plus élevé) est affecté à chaque JS. Si un agent est déjà affecté à une autre JS dans la même simulation, il est exclu des suivantes.

Lecture des résultats multi-JS

  • Taux de couverture : % des JS couvertes dans le scénario
  • Affectations par agent : liste des JS attribuées à chaque agent
  • JS non couvertes : JS restantes sans candidat disponible, avec gravité (OBLIGATOIRE vs DERNIER_RECOURS)
  • Détail des règles : même niveau de détail que la simulation simple, par affectation

Interface de simulation — Guide d'utilisation

Comment naviguer dans les résultats et interpréter chaque élément

Lancer une simulation JS simple

1
Ouvrir la fiche d'un agent absent
Depuis Agents → cliquer sur l'agent concerné.
2
Cliquer sur la JS à couvrir dans le planning
La JS imprévisible s'affiche en surbrillance. Le panneau d'analyse s'ouvre à droite.
3
Configurer le contexte (optionnel)
  • Imprévu partiel : si l'absence ne commence pas à l'heure prévue
  • Remplacement : cocher si l'agent qui vient remplace (seuils TE/repos assouplis)
  • Commentaire : note libre pour traçabilité
4
Cliquer sur "Analyser l'imprévu"
Le moteur lance l'analyse et affiche les résultats dans les 4 onglets.
5
Basculer "Sans figeage / Avec figeage"
Si le mode "Avec figeage" apporte des agents supplémentaires, un badge "+N agents" l'indique.

Carte candidat — Détail des calculs

Cliquer sur "Détail des règles" dans une carte candidat ou dans un nœud de cascade pour voir :

🌙 Repos journalier

Durée disponible entre le dernier poste et l'imprévu. Date et horaires du dernier poste affichés.

⏱ Amplitude

Amplitude de la JS imprévisible + raison du seuil appliqué (déplacement, remplacement, nuit…)

📈 TE GPT

Décomposition JS par JS du cumul TE dans la GPT, + la JS simulée, = total GPT.

📅 Longueur GPT

Nombre de JS dans la GPT après injection de l'imprévu. Limite : 6 JS.

📆 Repos périodiques

RP avant et après la GPT simulée. Vérifie la conformité des transitions (simple/double/triple).

✗ Violations bloquantes

Liste des règles en violation avec les valeurs numériques (mesurée vs. limite).

Lancer une simulation multi-JS

1
Aller dans "Simulation multi-JS"
Dans le menu latéral, cliquer sur "Simulation multi-JS".
2
Sélectionner les JS à couvrir
Cocher chaque JS dans la liste. Filtrer par date, agent, code JS selon les besoins.
3
Lancer la simulation
Les 4 scénarios sont calculés simultanément. Le meilleur scénario est pré-sélectionné.
4
Analyser les scénarios
Comparer taux de couverture, affectations VIGILANCE, JS non couvertes, figeages. Choisir le scénario le plus adapté au contexte opérationnel.
